# 📰 예언자 일보 #1 — AI에게 조수를 붙여주는 법

> Anthropic이 이번 주에 공식 블로그에 중요한 글을 올렸어요. 먼저 읽어봤는데요, 이거 우리한테 꽤 중요해요 📖

---

## 한 줄 요약

**혼자서 모든 걸 하는 AI보다, 조수를 부릴 줄 아는 AI가 더 잘해요.**

---

## 무슨 얘기냐면

Claude Code로 큰 작업을 하면, AI의 "책상"(컨텍스트 윈도우)에 서류가 쌓여요. 자료 조사, 곁길, 중간 메모... 결국 맨 밑의 중요한 서류가 밀려 떨어져요.

서브에이전트는 **"별도 메모장을 가진 조수"**를 부르는 거예요. 조수는 자기 메모장에서 작업하고, 결과만 보고해요.

> 💡 비유: 변호사가 소송 준비할 때 — 판례 조사, 증거 정리, 답변서 작성을 혼자 다 하면 머리가 터지잖아요. "판례는 이 사무장이 조사하고, 결과만 보고해" — 이게 서브에이전트예요.

---

## 서브에이전트의 핵심 특성 4가지

| # | 특성 | 왜 좋은가 |
|---|---|---|
| 1 | 대화 이력 없이 새로 시작 | 편견 없이 깨끗한 시각 |
| 2 | 여러 조수가 동시에 병렬 실행 | 3명이 동시에 → 3배 빠름 |
| 3 | 각각 다른 권한 부여 가능 | 리뷰어는 읽기만, 구현자는 수정까지 |
| 4 | 자동 생성 or 명시적 지시 | 자동화 수준에 따라 선택 |

---

## 쓰면 좋은 상황 vs 쓰면 안 되는 상황

| ⚡ 쓰면 좋은 상황 | 이유 |
|---|---|
| 파일 10개 이상 읽어야 할 때 | 조수가 읽고 요약만 받기 |
| 서로 안 겹치는 작업 3개 이상 | 조수 3명이 동시에 |
| 내가 만든 결과물 검토 | 편견 없는 새 조수에게 리뷰 |
| 설계 → 구현 → 테스트 파이프라인 | 릴레이 바통처럼 단계별 |

| 🚫 쓰면 안 되는 상황 | 이유 |
|---|---|
| 2단계가 1단계 결과를 꼭 알아야 할 때 | 릴레이 중 바통 떨어짐 |
| 같은 파일을 여러 조수가 동시 수정 | 충돌! |
| 간단한 작업 | 편지 한 통에 비서 부를 필요 없음 |
| 조수가 너무 많을 때 | 선택지 과다 → 오히려 불안정 |
| 조수들끼리 대화가 필요할 때 | 서브에이전트는 서로 못 봄 |

> "작업이 10개 이상의 파일을 탐색해야 하거나, 3개 이상의 독립적인 작업을 포함한다면, 위임이 필요하다는 강한 신호다." — Anthropic

---

## 자동화 수준별 4단계

처음부터 복잡하게 안 해도 돼요. 단계별로 올라가면 돼요.

| 단계 | 방법 | 예시 |
|---|---|---|
| 1단계 | 자연어로 말하기 | "서브에이전트를 써서 이 코드를 보안 검토해줘" |
| 2단계 | 커스텀 에이전트 저장 | `.claude/agents/보안리뷰.md` 파일 만들기 |
| 3단계 | CLAUDE.md에 규칙 | "코드 리뷰 시 항상 별도 에이전트 생성" |
| 4단계 | 스킬 & 훅 | 자동 실행, Stop 훅으로 테스트 통과 강제 |

---

## 실전 패턴 4가지

<details>
<summary>▸ 패턴 1: 리서치 후 구현</summary>

서브에이전트에게 먼저 리서치 위임 → 정리된 요약 받기 → 메인에서 구현.
10개 파일 직접 읽으면 다 컨텍스트에 쌓이지만, 조수에게 시키면 인사이트만 돌아와요.

</details>

<details>
<summary>▸ 패턴 2: 병렬 수정</summary>

같은 패턴의 업데이트를 여러 파일에 적용할 때.
세 개의 API 파일을 세 조수가 동시에 처리.

</details>

<details>
<summary>▸ 패턴 3: 독립적 리뷰</summary>

읽기 전용 + 대화 이력 없는 조수로 코드 평가.
자기가 쓴 코드에 대한 편향 없이 객관적 피드백.

</details>

<details>
<summary>▸ 패턴 4: 파이프라인 워크플로우</summary>

API 설계 → 엔드포인트 구현 → 통합 테스트.
각 단계의 출력 파일이 다음 단계의 바통.

</details>

## 업계는 어떻게 보고 있나

| 출처 | 핵심 메시지 |
|---|---|
| Anthropic | "대화형으로 시작하고, 나중에 자동화하라" |
| Deloitte | 에이전트 프로젝트 40%+ 가 2027년까지 취소 예측 |
| IBM | 가장 어려운 건 라우팅이 아니라 상태 관리 |
| Epsilla | 진짜 가치는 병렬이 아니라 컨텍스트 관리 (토큰 90%+ 절감 사례) |

## 가장 중요한 한 마디

> "대화형으로 시작하고, 나중에 자동화하라." — Anthropic

자연어로 시켜보고 → 패턴이 보이면 → 그때 자동화. 이 순서예요.
